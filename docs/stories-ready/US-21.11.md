## User Story

```
En tant que visiteur du site en situation de handicap (utilisateur clavier),
je veux que le menu mobile reponde correctement a Escape (fermeture)
et piege le focus a l'interieur tant qu'il est ouvert,
afin de ne pas me retrouver prisonnier d'un menu apres navigation clavier
sur un smartphone ou un desktop sans souris.
```

## Contexte

### Findings QA 10/04/2026

Le commit `28a8186 fix(a11y): implement keyboard nav for mobile menu (Escape + focus trap)` a livre un fix d'accessibilite critique **sans aucun test E2E de non-regression**. Le risque : toute modification future du composant `Header` ou du mobile menu peut casser silencieusement l'Escape handler ou le focus trap.

Ce gap a ete identifie dans la matrice de tracabilite du 10/04 :
> "Mobile menu keyboard nav (commit 28a8186) — fix en prod mais aucun test E2E dedie. Risque de regression si le header est modifie."

### Pourquoi un test E2E (et pas unit)

1. Le focus trap est une interaction **browser-level** (gestion de `document.activeElement`, boucle de Tab/Shift+Tab) — un test unit avec jsdom ne reproduit pas le vrai comportement du focus.
2. La fermeture via Escape implique un event listener sur `document` — plus fiable a tester dans un vrai navigateur.
3. Le cout est minime : ajout de ~40 lignes dans `tests/e2e/keyboard-nav.spec.ts` existant.

### Pourquoi 1 pt et pas 0.5

Estimation Fibonacci — la plus petite valeur disponible est 1 pt. Le scope est chirurgical (1 fichier existant + 2 tests) mais **doit** etre traite comme une unite INVEST complete, avec AC, DoD, et commit conventionnel.

## Notes techniques / decisions

### Pattern Playwright focus trap

```typescript
test("focus stays trapped inside mobile menu", async ({ page }) => {
  await page.goto("/fr");
  await page.getByRole("button", { name: /menu/i }).focus();
  await page.keyboard.press("Enter");

  const mobileMenu = page.locator("#mobile-menu");
  await expect(mobileMenu).toBeVisible();

  // Tab through all focusable elements
  const focusableCount = await mobileMenu.evaluate((el) =>
    el.querySelectorAll('a, button, [tabindex]:not([tabindex="-1"])').length
  );

  for (let i = 0; i < focusableCount + 2; i++) {
    await page.keyboard.press("Tab");
  }

  // Assert focus is still inside
  const focusInside = await page.evaluate(() => {
    const menu = document.getElementById("mobile-menu");
    return menu?.contains(document.activeElement) ?? false;
  });
  expect(focusInside).toBe(true);
});
```

### Piege : timing d'ouverture

Le mobile menu utilise une transition CSS (bg-deep). Ne **jamais** attendre via `waitForTimeout`. Utiliser `expect(mobileMenu).toBeVisible({ timeout: 2000 })` ou une assertion CSS (deja pattern dans `accessibility.spec.ts`).

### Piege : race d'hydratation React SSR (decouverte Sprint 13)

US-21.1 a revele un pattern important : le bouton hamburger est rendu
en SSR et visible **avant** que React ait attache ses onClick handlers.
Un premier `keyboard.press("Enter")` peut etre perdu si le handler n'est
pas encore attache — comportement non deterministe selon le cold-start
du dev server.

**Solution obligatoire** :

```ts
// 1. Navigation rapide
await page.goto("/fr", { waitUntil: "domcontentloaded" });

// 2. Wait pour hydratation React (aria-expanded est rendered par React)
const hamburger = page.getByRole("button", { name: /menu/i });
await expect(hamburger).toHaveAttribute("aria-expanded", "false");

// 3. Click + verify dans un expect.toPass pour absorber la race
await expect(async () => {
  await hamburger.focus();
  await page.keyboard.press("Enter");
  await expect(hamburger).toHaveAttribute("aria-expanded", "true", {
    timeout: 500,
  });
}).toPass({ timeout: 5000 });
```

Ce pattern est **deja utilise dans `tests/e2e/accessibility.spec.ts`**
(ligne ~60) et doit etre repris tel quel dans `keyboard-nav.spec.ts`
pour ne pas reintroduire la flakiness.
