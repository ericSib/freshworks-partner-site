```
En tant que lead dev du projet freshworks-partner-site,
je veux un rapport de mutation score automatise sur les modules critiques
(src/lib/quiz/scoring.ts, src/lib/quiz/generate-pdf.ts, src/lib/validation.ts,
```

## Contexte

### Findings QA 10/04/2026

La DoD QA du projet exige explicitement : *"Mutation score >= 60% sur le nouveau code"* (checklist DoD QA, `.claude/scheduled-tasks/acceptance-traceability-check/SKILL.md`, section 3).

**Etat actuel** : aucun outil de mutation testing n'est configure. Le taux de coverage de ligne (75.31%) donne une fausse impression de securite : un test peut traverser une ligne sans asserter son comportement.

**Exemple concret** : les 5 nouveaux tests de `scoring.ts` ecrits en Sprint 7 (US-18.2) couvrent 89.28% de lignes, mais rien ne prouve qu'ils detecteraient un off-by-one sur la ponderation des dimensions. Le mutation testing simule ces mutations et mesure si au moins 1 test echoue.

### Pourquoi "Could" et pas "Must"

1. **Pre-requis** : Stryker est valuable **apres** avoir remonte la couverture des modules critiques (US-21.6, 21.8, 21.9). Lancer Stryker sur un module a 30% de coverage produit un rapport inutile.
2. **Sprint 14 deja dense** : capacity cible 10-13 pts, la priorite va aux dettes pre-existantes (US-21.6, 21.7, 21.8, 21.9).
3. **ROI differencie** : un mutation score de 60% sur `scoring.ts` (le coeur du quiz) est 10x plus utile que sur les composants UI. Pragmatisme : ciblage restreint.
4. **Cout run** : Stryker multiplie le temps de test par ~20-30x. Un setup naif peut rendre la CI inutilisable. Necessite un scope restreint ET une strategie `--incremental`.

### Alternative consideree

**Option rejetee** : ne pas mesurer le mutation score et continuer avec coverage lines seul. Refus car la DoD QA l'exige explicitement. L'absence de mesure = dette silencieuse.

**Option retenue** : Stryker en mode **targeted** (5 modules critiques) + **incremental** (ne muter que les fichiers changes sur une PR). Gain : temps de run acceptable, signal qualite sur le chemin critique.

## Notes techniques / decisions

### Choix Stryker vs alternative

**Retenu** : [Stryker Mutator](https://stryker-mutator.io/) pour JS/TS.
- Ecosysteme mature (depuis 2016)
- Plugin vitest officiel (`@stryker-mutator/vitest-runner`)
- Mode incremental integre
- Rapport HTML riche (drill-down par fichier/ligne)

**Rejete** : PITest (Java), Mull (C++), Cosmic Ray (Python) — hors ecosysteme JS.
**Rejete** : approche manuelle (git apply mutation diff) — non industrialisable.

### Config minimale (skeleton)

```javascript
// stryker.conf.mjs
/** @type {import('@stryker-mutator/api/core').PartialStrykerOptions} */
export default {
  packageManager: "npm",
  reporters: ["html", "progress", "dashboard"],
  testRunner: "vitest",
  coverageAnalysis: "perTest",
  mutate: [
    "src/lib/quiz/scoring.ts",
    "src/lib/quiz/generate-pdf.ts",
    "src/lib/validation.ts",
    "src/lib/rate-limit.ts",
    "src/hooks/useQuiz.ts",
  ],
  thresholds: { high: 80, low: 60, break: 60 },
  incremental: true,
  incrementalFile: ".stryker-tmp/incremental.json",
  htmlReporter: { fileName: "reports/mutation/html/index.html" },
  vitest: {
    configFile: "vitest.config.ts",
  },
  timeoutMS: 30000,
  concurrency: 2, // reduit le load CPU en local
};
```

### Strategie d'extension du scope

**Phase 1** (US-21.10) : 5 modules critiques uniquement. Baseline >= 60%.
**Phase 2** (futur backlog) : ajout de `src/app/api/contact/route.ts` + `src/lib/analytics.ts` apres que leur coverage depasse 80%.
**Phase 3** : considerer les hooks UI (useScrollSpy, useContactForm) si le cout CI reste acceptable.

### Cout run attendu

| Scope | Temps local (M2) | Temps CI (ubuntu-latest) |
|---|---|---|
| 5 modules x ~100 lignes | ~3-5 min | ~6-10 min |
| +2 modules phase 2 | ~5-8 min | ~10-15 min |

Seuil de bascule "trop lent" : > 15 min. A ce moment-la, passer en incremental only (mute uniquement les fichiers modifies dans la PR).

### Piege : survived mutants false positives

Certaines mutations sont **equivalentes** (comportement identique, exemple : changer un `>` en `>=` sur une condition qui n'est jamais exactement egale). Ces cas peuvent gonfler artificiellement le mutation score a la baisse. Le fichier `stryker.ignores.md` documente chaque ignore avec justification.

### Non-objectif

Cette story **n'ameliore pas** la qualite des tests existants. Elle fournit un **outil de mesure**. L'amelioration suite aux findings sera traitee dans des stories distinctes (exemple : "US-21.12 kill survived mutants in scoring.ts").

### Runbook apres livraison

1. Lancer `npm run test:mutation` localement → noter le score baseline
2. Partager le rapport HTML avec le stakeholder
3. Creer US-21.12+ pour chaque module ou le score < 60% post-baseline
4. Integrer le check au sprint planning : toute nouvelle story touchant un module mute doit maintenir le score

### Suite logique

Apres US-21.10 :
- **Sprint 16+** : kill des survived mutants identifies (stories dediees)
- Passer le workflow mutation en **bloquant** sur `main` apres 2 sprints de stabilisation
