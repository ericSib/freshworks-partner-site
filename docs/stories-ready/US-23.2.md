## User Story

```
En tant que developpeur ou contributeur a11y maintenant le Header,
je veux que la logique du mobile menu et du focus trap soit extraite dans un composant et un hook dedies,
afin de reduire la complexite de Header.tsx (242L → ~120L) et de pouvoir tester le focus trap en isolation.
```

---

## Contexte

Detection par la tache planifiee `refactoring-radar` du 25/04/2026 :
- `src/components/layout/Header.tsx` : 242 lignes, **26 conditionnels**, 8 fonctions
- Trois preoccupations melees : navigation desktop, mobile menu (toggle + animation + focus trap), scroll detection
- Quadrant Fowler : dette **prudente/involontaire** (accumulee progressivement par ajouts successifs)

Le focus trap est aujourd'hui implemente inline avec des handlers keyboard ad hoc. Une extraction permettra de le reutiliser sur d'autres surfaces (modale Calendly future, drawer pricing, etc.).

Source : `docs/refinement/refactoring-radar-2026-04-25.md` §3 US-23.2.

## Notes techniques / decisions

### Pattern Fowler applique

- **Extract Component** sur MobileMenu
- **Extract Custom Hook** sur useFocusTrap

### Direction des dependances (skill `dev-architecture`)

```
Header.tsx
  ├─→ MobileMenu.tsx       (composition descendante)
  │     └─→ useFocusTrap   (hook utilitaire generique)
  └─→ navigation config    (reuse existant)
```

`useFocusTrap` n'a **aucune dependance projet** — hook generique, reutilisable Modal/Drawer futurs.

### Pourquoi 2 commits

1. `useFocusTrap` peut etre teste et merge **avant** d'etre utilise (pure addition)
2. `MobileMenu` extract reduit Header.tsx — verification de non-regression isolee

Effort 3 pts > 4 pts buffer (1 pt restant apres injection des 3 quick wins). Reportage S20 acte au refinement Radar.
