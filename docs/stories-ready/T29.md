## User Story

```
En tant que developpeur du projet freshworks-partner-site,
je veux que le pre-commit hook execute `npx tsc --noEmit` en plus du lint actuel,
afin de bloquer toute erreur TypeScript des le commit local et eviter
les cascades de drift TS detectees uniquement en CI.
```

---

## Contexte

Sprint 20 a connu une cascade de regression : 11 erreurs TS accumulees pendant les 5 runs CI rouges sur lockfile drift. Une fois le `npm ci` debloque, le step Type-check a remonte les erreurs en cascade. Le commit `794683e` (lint cleanup) a meme reintroduit une regression `useScrollReveal` qui aurait ete bloquee a la source par un pre-commit `tsc --noEmit`.

D41 acte le principe : **le pre-commit doit garantir un commit type-clean**. Cout 5-15s par commit, mitigation via `incremental: true` (deja active dans `tsconfig.json`).

## Criteres d'acceptation (Gherkin)

### Scenario 1 : Hook execute tsc en plus du lint

```gherkin
Etant donne le hook .husky/pre-commit existant qui execute "next lint --quiet"
Quand on inspecte le contenu du fichier
Alors le hook contient "npx tsc --noEmit" apres "next lint --quiet"
  ET les deux commandes echouent independamment (un fail bloque le commit)
```

### Scenario 2 : Drift TS bloque le commit

```gherkin
Etant donne un fichier source avec une erreur TS deliberee (ex: let x: number = "string")
Quand on tente git commit -m "test"
Alors le commit est refuse
  ET le message d'erreur contient le chemin du fichier + le numero de ligne
  ET le code de sortie est non-zero
```

### Scenario 3 : Code type-clean passe le hook

```gherkin
Etant donne un projet sans erreur TS
Quand on tente git commit -m "test"
Alors le commit est accepte
  ET la duree totale du hook < 30s sur le projet S21 actuel
```

### Scenario 4 : Pas de regression sur le hook lint existant

```gherkin
Etant donne un fichier avec un warning lint deliberre
Quand on tente git commit -m "test"
Alors le commit est refuse par le step lint avant meme d'atteindre tsc
  ET le message d'erreur mentionne ESLint
```

## Notes techniques / decisions

### Pourquoi `npx tsc --noEmit` et pas `tsc --noEmit`

Le `npx` garantit l'utilisation de la version locale (`node_modules/.bin/tsc`) plutot qu'une version globale qui pourrait etre desynchronisee.

### Pourquoi pas `lint-staged` + `tsc-files`

`tsc-files` execute `tsc` uniquement sur les fichiers modifies, mais cela cree des faux negatifs (un fichier non-modifie peut casser parce qu'un de ses imports a change). On execute `tsc --noEmit` sur tout le projet → cohenrent avec ce que CI fait.

### Cout attendu

`tsconfig.json` a deja `incremental: true` → la 1ere execution genere `tsbuildinfo`, les suivantes sont incrementales (~5-10s sur le projet S21 actuel). Cout acceptable.

### Risque : commit en urgence

Si le PO veut commiter en urgence avec une erreur TS connue, le hook bloque. Mitigation : `git commit --no-verify` reste possible (comme pour lint actuel). Documente dans `docs/PROCESS.md` §7.3.
