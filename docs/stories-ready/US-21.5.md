## Notes techniques

**Approche structurale vs visuelle** : on ne verifie PAS le rendu visuel du PDF (snapshot d'image serait fragile, lent, et hors-scope vitest). On verifie que les **bonnes primitives jsPDF** sont appelees avec les **bons parametres logiques** (dimensions, textes, nombre d'iterations).

**Piege mocking** : jsPDF est importe par `import jsPDF from "jspdf"` (default export). Le mock vitest doit exposer un constructor :

```typescript
vi.mock("jspdf", () => {
  const mockInstance = {
    setFillColor: vi.fn(),
    rect: vi.fn(),
    // ... toutes les methodes utilisees
    splitTextToSize: vi.fn((text: string) => [text]), // doit retourner un array
    save: vi.fn(),
  };
  return { default: vi.fn(() => mockInstance) };
});
```

**Non-objectif** : ce test ne couvre PAS la verification que le PDF produit est lisible dans un lecteur PDF. Ce serait un test E2E avec headless Chrome, out-of-scope ici.

**Suite logique** : apres US-21.5 → US-21.7 (refacto `QuizResultsPreview`) peut etre lance sereinement, car le bouton download PDF sera couvert de bout en bout (tests unitaires PDF + E2E quiz).
