# US-21.3 : Scan axe-core du parcours quiz (5 etats cles)

> **Epic** : E21 — Consolider la suite E2E (stabilite, couverture, a11y, POM)
> **Priorite** : Must
> **Estimation** : 3 pts
> **Sprint cible** : Sprint 13 "Fiabiliser et couvrir"
> **Auteur** : PO (Claude Code)
> **Date** : 10/04/2026

---

## User Story

```
En tant que visiteur du site What A Service en situation de handicap (moteur, visuel,
cognitif), qui souhaite evaluer la maturite de ses services via le quiz ITSM ou CX,
je veux que chaque etape critique du parcours quiz respecte WCAG 2.2 niveau AA,
afin de pouvoir completer mon evaluation et recevoir mon rapport sans etre
exclu du funnel principal de generation de lead.
```

---

## Contexte

Le quiz ITSM/CX est le **lead magnet central** du Product Goal (~50-200 leads/mois).
Un defaut d'accessibilite sur le parcours quiz signifie :
1. Un risque **legal** (obligation EAA — European Accessibility Act — juin 2025)
2. Un risque **conversion** : exclusion d'une partie de la cible (~15% de la population)
3. Un risque **reputation** : incoherence avec le positionnement "Consulting, Done Right"

**Etat actuel** :
- 0 scan axe-core sur le parcours quiz
- Les 3 E2E quiz existants verifient uniquement le **happy path fonctionnel**
- Le commit `096cb5b` a resolu les violations color-contrast homepage, **mais rien
  n'a ete verifie sur le parcours quiz**. Cette dette est explicite dans le
  PRODUCT-BACKLOG.md (note hors-sprint E21 : "couvre partiellement US-21.3").

**5 etats cles a couvrir** :
| # | Etat | Route | Locator indicatif |
|---|------|-------|-------------------|
| A | Selecteur ITSM/CX | `/fr/quiz` phase initiale | `getByRole('button', { name: /Services IT|Support Client/i })` |
| B | Formulaire demographiques | `/fr/quiz` apres clic ITSM | `#quiz-company-size`, `#quiz-industry`, `#quiz-role` |
| C | Question active (dim. 3) | `/fr/quiz` milieu du flow | `button[aria-pressed]` |
| D | Resultats (pre-email-gate) | `/fr/quiz` fin de flow | `text=/100`, `svg[role='img']` (radar) |
| E | Email gate focus | `/fr/quiz` scroll sur email gate | `getByPlaceholder('votre@email.pro')` |

**Choix bilingue** : ce sprint couvre **FR uniquement**. La couverture EN sera
ajoutee dans une US future (US-21.3b) si besoin, une fois l'infra a11y quiz
stabilisee. Raison : les composants sont identiques FR/EN, la difference ne
concerne que le texte (couvert par les tests i18n existants, 373 tests).

Product Goal lie : "leads 15-30% conversion" implique que **tout visiteur**
puisse completer le quiz. Un blocage a11y sur l'etat B ou C annulerait
l'investissement Sprint 7-9 (quiz fondations).

## Criteres d'acceptation (Gherkin)

### Scenario 1 : Scan A — Selecteur ITSM/CX

```gherkin
Etant donne un visiteur qui navigue vers "/fr/quiz"
Quand la page a termine son chargement (`domcontentloaded`)
  ET les 2 boutons "Services IT" et "Support Client" sont visibles
Quand un scan axe-core est execute avec wcag2a + wcag2aa + wcag21a + wcag21aa
Alors aucune violation de niveau "critical" ou "serious" n'est remontee
  ET chaque carte (ITSM/CX) est atteignable au clavier (Tab)
  ET le focus est visible sur la carte active (outline >= 2px contraste >= 3:1)
```

### Scenario 2 : Scan B — Formulaire demographiques

```gherkin
Etant donne le visiteur a cliqué sur "Services IT"
Quand le formulaire demographiques est visible (label "Quelques informations")
Quand un scan axe-core est execute
Alors aucune violation critical ou serious n'est remontee
  ET chaque select a un `label` associe (aria-labelledby ou htmlFor)
  ET le bouton "Demarrer" a un `name` accessible
  ET le bouton est disabled tant que les 3 selects ne sont pas remplis
    (etat communique via aria-disabled)
```

### Scenario 3 : Scan C — Question active

```gherkin
Etant donne le visiteur a rempli les demographiques et clique "Demarrer"
  ET la question 1 est affichee (sans avoir rien clique)
Quand un scan axe-core est execute
Alors aucune violation critical ou serious n'est remontee
  ET les 5 options ont `aria-pressed="false"` par defaut
  ET le step indicator "Section 1 de 9" est lisible par un lecteur d'ecran
  ET la structure heading est correcte (h1 unique sur la page, h2 pour la question)
```

### Scenario 4 : Scan D — Ecran resultats (pre-email-gate)

```gherkin
Etant donne le visiteur a complete les 9 questions (flow automatise du test)
  ET l'ecran de resultats affiche "/100", "Level 3", le radar chart
Quand un scan axe-core est execute
Alors aucune violation critical ou serious n'est remontee
  ET le radar chart SVG expose `role="img"` + `aria-label` descriptif
    (e.g., "Radar chart : scores par dimension")
  ET les sections "Quick wins" sont annoncees via headings h3
  ET le label "/100" est accessible (pas uniquement visuel)
```

### Scenario 5 : Scan E — Email gate avec focus

```gherkin
Etant donne l'ecran resultats affiche avec l'email gate visible
Quand le visiteur met le focus dans le champ email
  ET un scan axe-core est execute
Alors aucune violation critical ou serious n'est remontee
  ET le champ email a un label associe (meme si placeholder-only interdit)
  ET le bouton "Debloquer" est accessible
  ET le message de disclaimer RGPD (si present) est annonce par un lecteur d'ecran
```

### Scenario 6 : Rapport d'erreur detaille en cas d'echec

```gherkin
Etant donne un scenario axe-core en echec
Quand le test Playwright fail
Alors le rapport liste chaque violation avec : [impact] id: description (N occurrences)
  ET le format est identique a celui utilise dans `accessibility.spec.ts`
  ET l'erreur inclut le selecteur CSS de chaque element incrimine pour faciliter le fix
```

## Notes techniques / decisions

### Strategie de scan : par etat vs. parcours complet

**Choix retenu** : un scan par etat (A, B, C, D, E) plutot qu'un seul scan
`afterEach`. Avantages :
- Rapport d'erreur plus precis (l'etat fautif est identifie immediatement)
- Etats intermediaires (demographics, question active) impossibles a scanner
  autrement qu'en "freeze frame"
- Alignement avec le pattern `accessibility.spec.ts` (scan par interaction)

### Tags WCAG couverts

Comme `accessibility.spec.ts` : `["wcag2a", "wcag2aa", "wcag21a", "wcag21aa"]`.
**WCAG 2.2** (tags `wcag22a`, `wcag22aa`) est volontairement exclu pour cette
story — upgrade a faire dans une US future apres validation de l'axe-core 4.8+.

### Cas des false positives oklch + backdrop-blur

Si le quiz utilise un header avec backdrop-blur (a verifier en implementation),
la regle `color-contrast` peut remonter des false positives. Documenter en
commentaire dans le test et utiliser `.disableRules(['color-contrast'])`
**uniquement sur l'element concerne** via `.include(...)`.

### Gating email — bypass pour test

L'ecran resultats affiche une preview (score + radar) **sans email** — c'est
cette vue pre-gate qui est scannee en D. Le submit email en E met le focus
dans le champ sans envoyer le formulaire (pour eviter pollution HubSpot).

### Extraction d'un helper commun

Si le code de completion du quiz est duplique entre `quiz.spec.ts`,
`quiz-a11y.spec.ts` et US-21.4 (POM future), envisager :

```ts
// tests/e2e/helpers/quiz.ts
export async function completeItsmQuiz(page, { strategy = 'neutral' } = {}) { ... }
```

Mais cette extraction est **hors scope** de cette story (appartient a US-21.4 POM).
Si la duplication devient insupportable, documenter la dette dans le commit body.
