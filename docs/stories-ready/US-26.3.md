## User Story

```
En tant que developpeur responsable de la chaine deps WaS,
je veux surveiller la chaine resend → svix → uuid jusqu'a ce qu'une version non-vulnerable soit disponible,
afin d'eliminer la vulnerabilite uuid <14.0.0 (buffer bounds check) sans introduire de regression majeure (downgrade resend = breaking).
```

---

## Contexte

Detection par la tache planifiee `security--accessibility-scan` du 26/04/2026 :
- Severite : 🟢 LOW/MOD — `uuid` <14.0.0 vulnerable a un manque de buffer bounds check (GHSA-w5hq-g745-h8pq)
- Chaine : `resend@6.10.0` → `svix@1.x` → `uuid@<14`
- Risque reel : tres faible. Impact uniquement si on utilise `uuid v3/v5/v6` avec un buffer fourni par l'utilisateur (cas non present dans le code WaS — `uuid` est utilise par svix pour generer des request IDs internes Resend).

**Pourquoi pas un `npm audit fix --force`** : la seule solution proposee par npm audit est de downgrader `resend` a 6.1.3, qui est :
- Une release plus ancienne (sortie avant 6.10.0)
- Un breaking change potentiel sur l'API Resend
- Un anti-pattern : on retrograde une dep stable pour patcher un risque tres marginal

Source : `docs/refinement/security-scan-2026-04-26.md` §3.

## Notes techniques / decisions

### Pourquoi 0 pt et pas une story dev

Story **explicitement de process** — pas de code a livrer, pas d'incrementation de valeur livrable au sens Scrum classique. Le travail consiste uniquement a :
- Continuer le scan hebdomadaire (deja en place)
- Re-evaluer la situation chaque semaine
- Declencher une story dev si conditions changent

C'est le meme pattern que **T21** (PO i18n review > 50 cles) et **T25** (Mini-refinement obligatoire ops > 30 min) — process formels avec 0 pt.

### Risque accepte

Le PO (Eric Sib) accepte le risque residuel uuid <14 :
- Surface d'attaque : nulle dans le code WaS direct
- Surface d'attaque indirecte : limitee a l'usage interne svix (genere des UUIDs pour Resend API calls)
- Pas d'input utilisateur transmis a uuid

Cette acceptation est documentee ici. En cas d'audit externe ou de modification du contexte (nouvelle utilisation directe d'uuid), la story doit etre re-evaluee.

### Alternative ecartee : forker svix

Forker `svix` pour patcher la dep uuid serait un anti-pattern (maintenance lourde, divergence du upstream, risque securitaire propre). Ecartee.

### Cycle de vie attendu

Estimation : la story sera fermee dans 4-12 semaines, quand resend releasera une version qui upgrade svix qui upgrade uuid (chaine de release standard). Pas d'action proactive necessaire.
