## User Story

```
En tant qu'utilisateur de lecteur d'ecran completant le quiz Service Maturity Index,
je veux que le champ email de la gate de resultats annonce explicitement son objectif (et non un generique "edit text"),
afin de comprendre quelle information saisir sans devoir relire le contexte alentour.
```

---

## Contexte

Detection par la tache planifiee `security--accessibility-scan` du 26/04/2026 :
- Fichier : `src/components/quiz/QuizEmailGate.tsx:42`
- Element : `<input type="email">` avec `placeholder` mais sans `<label>`, `aria-label`, ou `aria-labelledby`
- Norme : WCAG 2.1 niveau **AA** §1.3.1 "Info and Relationships" + §3.3.2 "Labels or Instructions"
- Impact : screen readers (NVDA, JAWS, VoiceOver) annoncent "edit text" generique sans contexte

Source : `docs/refinement/security-scan-2026-04-26.md` §3.

A noter : le formulaire **fonctionne** correctement (validation, autofill, submit), c'est uniquement le niveau de conformite **AA** qui est manque. Le niveau A est respecte via `type="email"` + `placeholder`.

## Notes techniques / decisions

### Pourquoi aria-label et pas un `<label>` visible

La maquette actuelle place le champ en mode "inline form" (input + bouton sur la meme ligne) avec un titre h3 et un placeholder explicite. Ajouter un `<label>` visible casserait le design existant. `aria-label` est la solution WCAG-compliant qui :
- Preserve le visuel actuel (pas de regression UI)
- Fournit un nom accessible aux technologies d'assistance
- Est testable via axe-core et lecteurs d'ecran

### Alternative envisagee : `<label>` visuallyhidden

Pattern alternative `<label className="sr-only">` (Tailwind) — plus "purement" semantique, mais aria-label est plus simple, equivalent fonctionnellement, et coherent avec d'autres patterns du projet. Choix : aria-label.

### Pourquoi 1 pt

3 fichiers modifies (composant + 2 i18n), 1 test E2E a etendre, scope absolument minimal.

### Pourquoi report S21 et pas injection S20

- Severite a11y = LOW (le formulaire fonctionne, niveau A respecte)
- Sprint 20 marge consommee par US-26.1 (vuln HIGH)
- Pas d'urgence reglementaire (RGAA/EAA n'est pas un risque immediat sur cette page)
- Permet de l'embarquer dans une revue a11y plus large en S21 (audit complet axe-core post-S20)

### Audit a11y plus large recommande

A signaler au PO pour Sprint 21 refinement : un audit axe-core systematique sur les 8 pages cles (home, services × 4, quiz × 3) pourrait reveler d'autres findings AA. Story candidate : "US-26.4 — Audit a11y complet 8 pages + remediation findings AA" (3-5 pts, S21+).
