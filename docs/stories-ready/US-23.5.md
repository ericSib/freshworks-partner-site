## User Story

```
En tant que developpeur ajoutant un nouveau parametre a la page de maturite (ex: ROI),
je veux que les 11 props eparpillees du composant `MaturityPage` soient regroupees en 3 objets de configuration cohesifs (content, diagnosis, context),
afin de pouvoir ajouter ou modifier un parametre sans casser tous les call sites.
```

## Notes techniques / decisions

### Pattern Fowler applique

**Introduce Parameter Object** — regroupement des parametres lies par leur cohesion semantique :
- `content` : tout ce qui est texte editorial
- `diagnosis` : tout ce qui constitue le diagnostic de maturite (problems + recommendations)
- `context` : tout ce qui est contexte d'execution (segment, locale, CTA cible, timeframe)

### Direction des dependances (skill `dev-architecture`)

Les types vivent dans `src/config/quiz/types.ts` (cote domaine), importes par le composant `src/components/sections/MaturityPage.tsx` (cote infrastructure UI). **Direction conforme** : les composants dependent du domaine, pas l'inverse.

### Pourquoi maintenant (S19 buffer)

1. Quick win 1 pt — fits buffer S19
2. **Synergie SMI** : SMI-roi va ajouter des props a MaturityPage. Faire le refactor d'abord = la story SMI-roi est plus simple a integrer.
3. Pas de risque visuel — refactor type-driven, snapshot test couvre la non-regression.
