# US-18.9 : API quiz dediee `/api/quiz/submit` (endpoint + Zod + HubSpot)

> **Epic** : E18 — Score de Maturite ITSM+CX (quiz lead gen)
> **Priorite** : Must
> **Estimation** : **4 pts** (initial 3 + 1 pt refactor composant client suite a decision D3)
> **Sprint cible** : Sprint 13 "Fiabiliser et couvrir"
> **Auteur** : PO (Claude Code)
> **Date** : 10/04/2026
> **Decision retenue** : **D3 Option A** — endpoint dedie `/api/quiz/submit` (vs. mutualisation `/api/contact`)

---

## User Story

```
En tant que responsable marketing/commercial chez What A Service,
je veux que chaque completion du quiz ITSM ou CX produise un lead structure
dans HubSpot (avec score, niveau de maturite, segment, dimensions faibles),
afin de pouvoir segmenter les campagnes de nurture et prioriser les leads
chauds sans confondre les leads "quiz" avec les leads "contact form".
```

---

## Contexte

### Bug critique decouvert le 10/04/2026

Lors du refinement, un **bug silencieux** a ete identifie dans l'implementation
actuelle du quiz :

**Fichier** : `src/components/quiz/QuizResultsPreview.tsx:65-75`
```ts
await fetch("/api/contact", {
  method: "POST",
  body: JSON.stringify({
    name: email.split("@")[0],
    email,
    company: results.demographics.industry || "—",
    challenge: `quiz-${results.segment}`,  // → "quiz-itsm" ou "quiz-cx"
    website: "",
  }),
});
```

**Schema Zod actuel** (`src/lib/validation.ts:39-41`) :
```ts
challenge: z.enum(["adoption", "cx", "tool", "migration", "scale", "itam", "other"])
```

Les valeurs `"quiz-itsm"` / `"quiz-cx"` **ne matchent aucune entree** de l'enum
→ la validation retourne **400 Bad Request** → `upsertHubSpotContact` n'est
jamais appele → **aucun lead quiz n'est capture dans HubSpot**.

Le `try/catch` cote client (ligne 76-78) **masque l'erreur** et affiche
`gateState === "unlocked"` quoiqu'il arrive. Cote utilisateur, l'experience
est transparente, mais **cote business, le lead magnet central est casse**.

**A verifier** : compter les leads HubSpot avec `challenge=quiz-*` sur les 30
derniers jours. Si zero, le bug est confirme.

### Pourquoi Option A (endpoint dedie)

Apres analyse comparative (voir decision D3 dans le refinement du 10/04/2026),
l'endpoint dedie a ete retenu pour :

1. **Separation des preoccupations** : contact form et quiz ont des modeles de
   donnees fondamentalement differents. Un polymorphisme force via discriminated
   union alourdirait le code pour peu de benefice.
2. **Observabilite** : un endpoint dedie permet des logs, metriques et events
   GA4 propres sans filtrage.
3. **Rate limiting independant** : un prospect peut soumettre contact form +
   quiz ITSM + quiz CX sans etre bloque par collision d'IP.
4. **Segmentation HubSpot** : les proprietes custom quiz (`quiz_score`,
   `quiz_segment`, etc.) permettent des filtres et nurture campaigns dedies.
5. **Alignement avec Phase 2** : futures routes `/api/scanner/*` (E19/E20)
   suivront le meme pattern.
6. **Extensibilite** : ajout futur du PDF, webhook HubSpot workflow, tracking
   conversion sans toucher au contact form.

### Impact business

Une fois cette story livree :
- **100% des completions quiz** produisent un lead exploitable dans HubSpot
- **Segmentation propre** : tableau de bord HubSpot filtrable par segment/level/score
- **Nurture differencie** : campagnes email adaptees au niveau de maturite
- **Priorisation commerciale** : les leads avec score >= 70 remontent en top

Product Goal lie : les **50-200 leads/mois** cibles sont impossibles a suivre
tant que ce bug n'est pas corrige.

```gherkin

  ET l'email de confirmation est envoye au prospect via Resend
```

### Scenario 2 : Validation Zod rejette un payload invalide

```gherkin
Etant donne un payload avec un `segment` invalide (e.g. "foo")
Quand la requete POST est envoyee a "/api/quiz/submit"
Alors la reponse est 400 Bad Request
  ET le body contient `{ error: "Validation failed", details: [...] }`
  ET aucun appel HubSpot n'est fait
  ET aucun email n'est envoye
```

Meme comportement pour :
- `email` absent ou malforme
- `overallScore` < 0 ou > 100
- `maturityLevel.level` en dehors de [1, 5]
- `dimensionScores` absent ou vide
- `demographics` manquant un champ requis

### Scenario 3 : Rate limiting independant du contact form

```gherkin
Etant donne une IP qui a deja soumis 5 fois "/api/contact" dans les 15 dernieres minutes
Quand elle envoie une requete a "/api/quiz/submit"

  ET le rate limit quiz est incremente de 1 (bucket independant)

Et :
Etant donne la meme IP qui envoie 11 fois "/api/quiz/submit" en 15 minutes (cap = 10)
Quand la 11eme requete arrive

```

### Scenario 5 : Fallback silencieux en cas d'erreur HubSpot/Resend

```gherkin
Etant donne un payload valide

Quand le handler traite la requete
```

## Notes techniques / decisions

### Structure du nouveau Zod schema

```ts
// src/lib/validation.ts (ajout)

export const QUIZ_SEGMENTS = ["itsm", "cx"] as const;
export const MATURITY_URGENCIES = ["high", "medium", "low"] as const;

export const maturityLevelSchema = z.object({
  level: z.number().int().min(1).max(5),
  labelKey: z.string().min(1),
  descriptionKey: z.string().min(1),
  urgency: z.enum(MATURITY_URGENCIES),
});

export const demographicsSchema = z.object({
  companySize: z.string().min(1),
  industry: z.string().min(1),
  role: z.string().min(1),
});

export const weakestDimensionSchema = z.object({
  dimensionId: z.string().min(1),
  score: z.number().min(0).max(100),
});

export const quizSubmitSchema = z.object({
  email: z.string().regex(EMAIL_REGEX).max(254),
  segment: z.enum(QUIZ_SEGMENTS),
  overallScore: z.number().min(0).max(100),
  maturityLevel: maturityLevelSchema,
  dimensionScores: z.record(z.string(), z.number().min(0).max(100)),
  demographics: demographicsSchema,
  weakestDimensions: z.array(weakestDimensionSchema).min(1).max(8),
});

export type QuizSubmitPayload = z.infer<typeof quizSubmitSchema>;
```

### Refactor du rate limiter

**Option proposee** (non obligatoire si trop long) : extraire en factory

```ts
// src/lib/rate-limit.ts

interface RateLimiterConfig {
  maxRequests: number;
  windowMs: number;
  keyPrefix?: string;
}

export function createRateLimiter(config: RateLimiterConfig) {
  const store: Map<string, number[]> = new Map();

  return {
    check(ip: string): RateLimitResult { ... },
    reset(): void { ... },
  };
}

// Conservation de l'API existante pour /api/contact
export const contactRateLimiter = createRateLimiter({
  maxRequests: 5,
  windowMs: 15 * 60 * 1000,
});

// Nouveau bucket pour le quiz
export const quizRateLimiter = createRateLimiter({
  maxRequests: 10,
  windowMs: 15 * 60 * 1000,
});

// Retrocompatibilite pour les tests existants
export const checkRateLimit = contactRateLimiter.check;
```

**Alternative plus simple** : garder `checkRateLimit(ip, bucket = 'contact')`
avec un second parametre. Moins elegant mais 0 refactor des tests existants.
