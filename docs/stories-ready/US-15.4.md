# US-15.4 : Executer Playwright dans GitHub Actions CI

> **Epic** : E15 — Pipeline CI/CD
> **Priorite** : **Must** (decision D4 tranchee le 10/04/2026, ex-Could)
> **Estimation** : 5 pts
> **Sprint cible** : Sprint 13 "Fiabiliser et couvrir"
> **Auteur** : PO (Claude Code)
> **Date** : 10/04/2026

---

## User Story

```
En tant que developpeur du projet What A Service,
je veux que la suite Playwright E2E (homepage + quiz + a11y + mentions-legales)
s'execute automatiquement a chaque pull request et push sur main,
afin de detecter les regressions fonctionnelles et d'accessibilite
avant le merge, sans dependre de l'execution locale manuelle.
```

---

## Contexte

### Etat actuel

- **GitHub Actions** existe (`.github/workflows/ci.yml`) et execute lint +
  typecheck + unit tests + build sur chaque PR/push main
- **Playwright** est configure (`playwright.config.ts`) avec :
  - 2 projects : `chromium` + `mobile-chrome`
  - `retries: 2` en CI (deja prevu)
  - `workers: 1` en CI (deterministe)
  - `webServer: npm run dev` (local) — **pas adapte a la CI**
- **11 tests E2E** aujourd'hui (8 homepage + 3 quiz), bientot **+10** apres
  livraison US-21.1, US-21.2, US-21.3 = ~21 tests
- **0 execution en CI** — les tests E2E ne tournent qu'en local

### Risques couverts

1. **Regression silencieuse** : les fix e2e du 09/04/2026 (`a300a20`,
   `05a2749`) ont ete necessaires car des regressions etaient passees
   inapercues sans execution CI
2. **Dette cognitive** : forcer chaque dev a lancer Playwright localement est
   inefficace et entraine des oublis
3. **Quality gate manquant** : le pipeline actuel laisse passer des PR avec
   une suite E2E cassee, ce qui degrade progressivement la couverture
4. **Deploiement a risque** : Vercel deploie automatiquement depuis main sans
   garantie E2E → un bug UI peut atteindre production

### Justification "Must" (D4 tranchee)

Initialement "Could" dans le backlog, cette story a ete remontee en **Must**
pour le Sprint 13 pour les raisons suivantes :

1. **Investissement deja realise** : US-21.1 + US-21.2 + US-21.3 ajoutent 10+
   tests E2E. Sans CI, ces tests sont ecrits mais jamais executes hors dev.
2. **Sprint theme "Fiabiliser"** : le Sprint Goal "fiabiliser le quiz en
   production" n'est pas coherent sans garantie d'execution systematique.
3. **Dette de traceabilite** : les fix hors-sprint recents (voir notes E21
   dans PRODUCT-BACKLOG.md) auraient ete detectes plus tot avec CI.
4. **Pre-requis de US-18.9** : l'ajout d'un endpoint `/api/quiz/submit` doit
   etre valide E2E avant deploiement — CI obligatoire.

Product Goal lie : la **"machine de generation de leads"** vise 50-200 leads/mois.
Une seule regression UI silencieuse peut couter des dizaines de leads. Le ROI
de la CI est immediat.

## Criteres d'acceptation (Gherkin)

### Scenario 1 : Workflow E2E declenche sur chaque PR

```gherkin
Etant donne une pull request ouverte vers main
Quand GitHub Actions est declenche
Alors le job "e2e" s'execute en parallele de "ci" (lint/test/build)
  ET le job "e2e" apparait dans la checks list de la PR
  ET le statut du job est visible dans l'UI GitHub avant merge
```

### Scenario 2 : Workflow E2E declenche sur push main

```gherkin
Etant donne un push direct (ou un merge) sur la branche main
Quand le workflow CI est declenche
Alors le job "e2e" s'execute avec la meme configuration que sur PR
```

### Scenario 3 : Installation des navigateurs Playwright

```gherkin
Etant donne un runner ubuntu-latest fresh
Quand le job "e2e" commence
Alors les navigateurs Playwright sont installes via `npx playwright install --with-deps chromium`
  ET le cache des browsers (`~/.cache/ms-playwright`) est utilise si disponible
  ET la duree totale d'installation est <= 90 secondes (avec cache hit)
```

### Scenario 4 : Build production + serveur de test

```gherkin
Etant donne le job "e2e" en cours
Quand les tests doivent tourner
Alors l'application est build via `npm run build`
  ET un serveur production est demarre via `npm run start` (PAS `npm run dev`)
  ET le serveur ecoute sur http://localhost:3000
  ET Playwright attend le serveur pret via `webServer` config
  ET les tests s'executent sur la build optimisee (pas le dev mode)
```

### Scenario 5 : Execution des tests sur chromium + mobile-chrome

```gherkin
Etant donne le projet `chromium` et le projet `mobile-chrome` configures
Quand `npx playwright test` est execute
Alors les 2 projects sont lances (matrice Playwright interne)
  ET la commande utilise `--reporter=list,html`
  ET les retries sont limitees a 2 (config existante)
  ET workers = 1 (deterministe)
```

### Scenario 6 : Echec bloque le merge

```gherkin
Etant donne une PR dont au moins 1 test Playwright echoue
Quand le job "e2e" termine
Alors son exit code est != 0
  ET le check GitHub affiche "Failed"
  ET si branch protection "require status checks" est activee sur main,
    le merge est bloque jusqu'a correction
```

### Scenario 7 : Artefacts de debug en cas d'echec

```gherkin
Etant donne un test Playwright qui fail
Quand le job "e2e" termine avec echec
Alors le dossier `playwright-report/` est uploade comme artefact GitHub
  ET le dossier `test-results/` est uploade (screenshots + traces + videos)
  ET les artefacts sont conserves 7 jours
  ET les artefacts sont telechargeables depuis l'UI GitHub Actions
```

### Scenario 8 : Variables d'environnement secretes

```gherkin
Etant donne le job "e2e" qui demarre le serveur de prod
Quand l'environnement est prepare
Alors les secrets suivants sont injectes via GitHub Secrets :
  | Variable              | Source                    |
  | RESEND_API_KEY        | Secret (OU stub vide)     |
  | HUBSPOT_ACCESS_TOKEN  | Secret (OU stub vide)     |
  | NEXT_PUBLIC_GA_ID     | Secret (OU stub vide)     |
  | CONTACT_EMAIL         | Variable publique          |
  ET aucun secret n'apparait dans les logs
  ET si un secret n'est pas defini, le serveur demarre quand meme
    (les handlers ont un fallback en dev mode)
```

**Note importante** : pour minimiser la surface d'attaque, les tests E2E doivent
tourner avec des **stubs vides** (pas de vrais tokens) sauf si explicitement
necessaire. Le contact form mock deja `api/contact` dans les tests existants.

### Scenario 9 : Performance du pipeline

```gherkin
Etant donne le workflow e2e execute en mode normal (sans retry)
Quand il termine
Alors la duree totale est <= 8 minutes (cold cache) ou <= 5 minutes (cache hit)
  ET le workflow global (ci + e2e en parallele) reste <= 10 minutes
  ET la duree est reportee dans les logs GitHub Actions
```

### Scenario 10 : Cache NPM + cache Playwright browsers

```gherkin
Etant donne 2 executions consecutives du workflow sur le meme package-lock.json
Quand la seconde execution demarre
Alors les dependances NPM sont restaurees depuis le cache (actions/setup-node cache)
  ET les navigateurs Playwright sont restaures depuis `~/.cache/ms-playwright`
  ET le gain de temps sur la 2eme execution est d'au moins 60 secondes
```

## Notes techniques / decisions

### Choix : job separe vs. job unique

**Retenu** : job **separe** `e2e.yml` plutot que d'ajouter un step au job `ci` existant.

| Raison | Detail |
|--------|--------|
| Parallelisation | `ci` (lint/test/build) + `e2e` (playwright) tournent en parallele → duree globale reduite |
| Isolation des secrets | Seul le job `e2e` a besoin des tokens HubSpot/Resend |
| Cache distinct | Cache Playwright browsers independant du cache npm |
| Lisibilite | Chaque job a un role clair, l'onglet Actions est plus lisible |
| Rerun ciblé | On peut relancer uniquement `e2e` sans refaire `ci` |

**Inconvenient** : duplication du checkout + setup node (acceptable, ~15s).

### Choix : npm run start vs. npm run dev en CI

**Retenu** : `npm run start` apres `npm run build`.

| Raison | Detail |
|--------|--------|
| Perf | Mode production = code optimise, plus rapide au runtime |
| Fidelite | Les tests doivent tourner contre ce qui sera deploye |
| Detection bugs prod | Certains bugs (ex: CSP stricte) n'apparaissent qu'en prod |
| Pas de HMR | Aucun overhead de hot reload en CI |

**Compromis** : le cold start est un peu plus lent (~10s), mais largement
compense par la performance en runtime.

### Skeleton du workflow

```yaml
# .github/workflows/e2e.yml
name: E2E

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

concurrency:
  group: e2e-${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  e2e:
    name: Playwright E2E
    runs-on: ubuntu-latest
    timeout-minutes: 15
    env:
      RESEND_API_KEY: ${{ secrets.RESEND_API_KEY }}
      HUBSPOT_ACCESS_TOKEN: ${{ secrets.HUBSPOT_ACCESS_TOKEN }}
      NEXT_PUBLIC_GA_ID: ${{ secrets.NEXT_PUBLIC_GA_ID }}

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - name: Install dependencies
        run: npm ci

      - name: Get Playwright version
        id: playwright-version
        run: echo "version=$(npm list @playwright/test --json | jq -r '.dependencies["@playwright/test"].version')" >> $GITHUB_OUTPUT

      - name: Cache Playwright browsers
        uses: actions/cache@v4
        id: playwright-cache
        with:
          path: ~/.cache/ms-playwright
          key: playwright-${{ runner.os }}-${{ steps.playwright-version.outputs.version }}

      - name: Install Playwright browsers
        if: steps.playwright-cache.outputs.cache-hit != 'true'
        run: npx playwright install --with-deps chromium

      - name: Install Playwright OS dependencies (always, even with cache hit)
        if: steps.playwright-cache.outputs.cache-hit == 'true'
        run: npx playwright install-deps chromium

      - name: Build
        run: npm run build

      - name: Run Playwright tests
        run: npx playwright test --reporter=list,html

      - name: Upload Playwright report
        if: failure()
        uses: actions/upload-artifact@v4
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 7

      - name: Upload test results (traces, screenshots, videos)
        if: failure()
        uses: actions/upload-artifact@v4
        with:
          name: test-results
          path: test-results/
          retention-days: 7
```

### Modification de playwright.config.ts

```diff
  webServer: {
-   command: "npm run dev",
+   command: process.env.CI ? "npm run start" : "npm run dev",
    url: "http://localhost:3000",
    reuseExistingServer: !process.env.CI,
-   timeout: 30_000,
+   timeout: 60_000,
  },
```

Raison du `timeout: 60_000` : `npm run start` peut etre plus lent a demarrer
qu'un dev server avec code deja compile.

### Strategie de cache Playwright

**Cle de cache** : `playwright-${runner.os}-${version}` (lie a la version
dependance). Un upgrade de `@playwright/test` invalide automatiquement le cache.

**Hit rate attendu** : ~95% (version Playwright change rarement entre PRs).
Gain moyen : 60-90 secondes par run.

### Secrets minimaux pour demarrer

Pour eviter de bloquer la mise en place, on peut initialement lancer la CI
**sans aucun secret configure**. Les handlers ont un mode dev :
- `RESEND_API_KEY` absent → `console.log` + skip Resend
- `HUBSPOT_ACCESS_TOKEN` absent → `console.log` + skip HubSpot
- `NEXT_PUBLIC_GA_ID` absent → pas de tracking GA4

Les tests E2E existants sont deja concus pour tourner sans ces secrets (ils
mockent `/api/contact` et verifient le UI state, pas les effets de bord).

### Risques et mitigations

| Risque | Mitigation |
|--------|------------|
| Tests flaky sur runner lent | US-21.1 deploye avant (elimine waitForTimeout) |
| Duree pipeline trop longue | Cache Playwright + job parallele a `ci` |
| Cout GitHub Actions | Sur repo public : gratuit. Sur repo prive : ~4 min/run × 30 runs/mois = 120 min gratuits compris dans le plan |
| Fuite de secrets | Stubs vides par defaut + revue du workflow en PR |
| Cache cassé apres upgrade | Cle de cache liee a la version (regeneree automatiquement) |
| Playwright nouvelles versions | Dependabot ouvre des PR → CI valide avant merge |
