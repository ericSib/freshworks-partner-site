# US-21.2 : Test E2E smoke + a11y sur /mentions-legales (FR+EN)

> **Priorite** : Should
> **Estimation** : 2 pts
> **Sprint cible** : Sprint 13 "Fiabiliser et couvrir"
> **Auteur** : PO (Claude Code)
> **Date** : 10/04/2026

---

## User Story

```
En tant que visiteur du site What A Service (en particulier en situation de handicap)
qui a besoin d'acceder aux mentions legales pour verifier la conformite RGPD,
je veux que la page /mentions-legales soit testee automatiquement
en termes de contenu (smoke) et d'accessibilite (axe-core) en FR et EN,
afin que toute regression sur cette page obligatoire soit detectee avant le deploiement.
```

---

## Contexte

La page `/mentions-legales` est une exigence **legale** (conformite RGPD + LCEN
en France) et represente un **blocker de confiance** pour les prospects B2B
(DSI Sophie, Marc, Karim, Nadia) qui verifient systematiquement l'existence
d'une entite legale avant d'engager un achat.

**Etat actuel** :
- Page existante : `src/app/[locale]/mentions-legales/page.tsx`
- Contenu : `src/app/[locale]/mentions-legales/LegalContent.tsx` (50 lignes, 6 sections h2)
- Breadcrumb : ajoute en Sprint 11 (`Breadcrumb locale={locale} slug="mentions-legales"`)
- i18n : cle `legal.*` dans `src/messages/{fr,en}.json`
- **Couverture E2E** : **0 test** — aucune regression detectable automatiquement
- **Couverture a11y** : aucun scan axe-core dedie

**Structure de la page** :
```
/mentions-legales
 ├── Breadcrumb (Accueil → Mentions legales)
 ├── <h1>Mentions legales</h1>
 └── 6 sections <h2> :
      ├── Editeur du site
      ├── Hebergement
      ├── Propriete intellectuelle
      ├── Donnees personnelles & RGPD
      ├── Cookies
      └── Contact
```

**Risque couvert** : une modification de `LegalContent.tsx` ou des cles i18n
`legal.*` qui casserait le rendu ne serait detectee qu'en production.

Product Goal lie : "production-grade" implique que toutes les pages obligatoires
soient testees, pas uniquement les pages de conversion.

## Criteres d'acceptation (Gherkin)

### Scenario 1 : Smoke FR — structure de la page

```gherkin
Etant donne un visiteur qui navigue vers "/fr/mentions-legales"
Quand la page a termine son chargement (`domcontentloaded`)
Alors un element h1 est visible avec le texte contenant "Mentions legales"
  ET exactement 6 elements h2 sont presents
  ET ces h2 contiennent respectivement : "Editeur", "Hebergement",
    "Propriete intellectuelle", "Donnees personnelles", "Cookies", "Contact"
  ET le composant Breadcrumb est visible en haut de la page
  ET le lien "Accueil" du breadcrumb pointe vers "/fr"
```

### Scenario 2 : Smoke EN — equivalent anglais

```gherkin
Etant donne un visiteur qui navigue vers "/en/mentions-legales"
Quand la page a termine son chargement
Alors un element h1 est visible (texte en anglais attendu)
  ET 6 sections h2 sont presentes
  ET le breadcrumb affiche "Home → ..." (traduction EN)
  ET aucun texte francais n'est visible dans la zone de contenu (regex test)
```

### Scenario 3 : Navigation — depuis le footer vers /mentions-legales

```gherkin
Etant donne un visiteur sur "/fr"
Quand il clique sur le lien "Mentions legales" du footer
Alors il est redirige vers "/fr/mentions-legales"
  ET la page /mentions-legales s'affiche correctement (scenario 1 re-valide)
```

### Scenario 4 : A11y axe-core FR — 0 violation critical/serious

```gherkin
Etant donne la page "/fr/mentions-legales" chargee
Quand un scan axe-core est execute avec les tags wcag2a + wcag2aa + wcag21a + wcag21aa
Alors aucune violation de niveau "critical" n'est remontee
  ET aucune violation de niveau "serious" n'est remontee
  ET en cas d'echec, le rapport liste chaque violation avec son impact, son id
    et le nombre d'occurrences (format identique a accessibility.spec.ts existant)
```

### Scenario 5 : A11y axe-core EN — meme exigence

```gherkin
Etant donne la page "/en/mentions-legales" chargee
Quand un scan axe-core est execute
Alors aucune violation critical ou serious n'est remontee
```

## Notes techniques / decisions

### Pattern de reference

Reprendre le pattern de `tests/e2e/accessibility.spec.ts` pour les scans axe-core :

```ts
const results = await new AxeBuilder({ page })
  .withTags(["wcag2a", "wcag2aa", "wcag21a", "wcag21aa"])
  .analyze();

const critical = results.violations.filter(
  (v) => v.impact === "critical" || v.impact === "serious"
);
```

### Cas de false positive connu (oklch + backdrop-blur)

Le fichier `accessibility.spec.ts` desactive `color-contrast` sur le mobile menu
(ligne 76) en raison d'un bug d'axe 4.x avec les couleurs oklch derriere un
backdrop-blur. La page `/mentions-legales` n'utilise pas de backdrop-blur —
**aucune desactivation de regle ne devrait etre necessaire**.

Si un false positive apparait malgre tout, documenter le dans un commentaire
de code avec la reference au bug axe-core en question.

### Couverture visuelle (hors scope)

Les tests visuels/screenshots ne sont PAS inclus. Cette story reste focalisee
sur smoke + a11y. Un test visuel pourra etre ajoute ulterieurement (hors E21).

### Performance attendue

Page statique simple → duree test < 3 secondes FR + 3 secondes EN + 2 scans axe ~8 secondes total.
