## Contexte

### Findings QA 10/04/2026

Le module `src/lib/hubspot.ts` est actuellement couvert a seulement **28.12% statements / 29.03% lines / 14.28% branches** (rapport vitest coverage du 10/04).

| Fonction exportee | Couverte ? |
|---|---|
| `splitName(fullName)` | ✅ (via `hubspot.test.ts` existant) |
| `buildProperties(data)` | ✅ partielle (happy path) |
| `upsertHubSpotContact(data)` | ❌ **0% branches** (lignes 141-168 non couvertes) |

Les fonctions internes non exportees `findContactByEmail`, `createContact`, `updateContact` contiennent toute la logique reseau + error handling et ne sont testees par **aucune assertion**.

### Pourquoi c'est critique

1. **Canal contact** : le formulaire `/api/contact` reste le 2e canal de capture de lead apres le quiz. Une regression silencieuse (typo propriete HubSpot, code HTTP mal traite, rename `lifecyclestage`) casserait le pipeline sans alerter.
2. **Bug historique** : le pattern `try/catch + console.error + return` de `upsertHubSpotContact` a deja masque le bug `challenge="quiz-*"` pendant des semaines (confirme par audit 10/04, corrige Sprint 13 via US-18.9). Les tests doivent couvrir explicitement les branches d'erreur pour qu'une future regression leve bien une alerte de coverage.
3. **Coherence avec le module quiz** : `src/lib/quiz/hubspot.ts` (livre US-18.9) est deja a 100% fonctions / ~78% branches. Aligner `src/lib/hubspot.ts` (historique) sur le meme niveau elimine la divergence de dette entre les 2 modules.

### Non-scope (deja couvert par US-18.9)

`src/lib/quiz/hubspot.ts` et `buildQuizProperties` sont **hors scope** de cette story — ils ont ete livres avec leur propre suite de tests dans US-18.9 Sprint 13 (`src/lib/quiz/__tests__/hubspot.test.ts`, 8 tests).

## Notes techniques / decisions

### Strategie mocking

**Pattern retenu** : mock direct de `global.fetch` via `vi.fn()`. Rejete : MSW (overkill pour 2 endpoints, ajout de dependance non justifie).

```typescript
beforeEach(() => {
  global.fetch = vi.fn();
  vi.stubEnv("HUBSPOT_ACCESS_TOKEN", "fake-token-123");
});

afterEach(() => {
  vi.unstubAllEnvs();
  vi.restoreAllMocks();
});

// Exemple assertion multi-call
it("searches then creates on not-found branch", async () => {
  (global.fetch as Mock)
    .mockResolvedValueOnce({ ok: true, json: async () => ({ total: 0, results: [] }) })
    .mockResolvedValueOnce({ ok: true, json: async () => ({ id: "new-42" }) });

  await upsertHubSpotContact(validPayload);

  expect(global.fetch).toHaveBeenCalledTimes(2);
  expect(global.fetch).toHaveBeenNthCalledWith(
    1,
    expect.stringContaining("/crm/v3/objects/contacts/search"),
    expect.objectContaining({ method: "POST" })
  );
  expect(global.fetch).toHaveBeenNthCalledWith(
    2,
    expect.stringMatching(/\/crm\/v3\/objects\/contacts$/),
    expect.objectContaining({ method: "POST" })
  );
});
```

### Piege : upsertHubSpotContact est async ET silent

Les tests doivent **await** explicitement la promesse et verifier que **ni resolve ni reject** ne sont bloquants. Pattern :

```typescript
await expect(upsertHubSpotContact(payload)).resolves.toBeUndefined();
```

### Non-objectif

On ne teste PAS :
- La disponibilite reelle de l'API HubSpot (test E2E = hors-scope, externe)
- Le format exact de la reponse HubSpot (couvert par le contrat HubSpot v3 stable)
- Les rate limits HubSpot (hors-scope, gere par retry infra cote HubSpot)

### Suite logique

Apres US-21.8 → US-18.9 peut cloner `upsertHubSpotQuizLead` avec confiance. Le pattern de tests devient reutilisable pour le nouveau module `src/lib/quiz/hubspot.ts`.
