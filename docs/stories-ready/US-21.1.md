# US-21.1 : Eliminer waitForTimeout dans la suite E2E

> **Epic** : E21 — Consolider la suite E2E (stabilite, couverture, a11y, POM)
> **Priorite** : Must
> **Estimation** : 1 pt
> **Sprint cible** : Sprint 13 "Fiabiliser et couvrir"
> **Auteur** : PO (Claude Code)
> **Date** : 10/04/2026

---

## User Story

```
En tant que developpeur maintenant la suite E2E Playwright,
je veux remplacer tous les appels `page.waitForTimeout(ms)` par des attentes
explicites basees sur l'etat de l'UI (visibilite, texte, URL, role ARIA),
afin de garantir 0 flakiness en CI et des tests 100% deterministes,
independants du load machine et du debit reseau.
```

---

## Contexte

Lors du refinement du 10/04/2026, 3 occurrences residuelles de `waitForTimeout`
ont ete identifiees :

- `tests/e2e/quiz.spec.ts:31` — attente 500ms apres clic option (auto-advance)
- `tests/e2e/quiz.spec.ts:82` — attente 500ms apres clic option (auto-advance)
- `tests/e2e/accessibility.spec.ts:68` — attente 600ms apres ouverture mobile menu

Ces attentes fixes sont un anti-pattern Playwright bien documente :
> `waitForTimeout` is discouraged because it makes tests flaky and slow.
> — [Playwright docs](https://playwright.dev/docs/actionability)

**Source interne** (`src/hooks/useQuiz.ts:92,112`) :
- Auto-advance entre questions : `setTimeout(350)`
- Transition questions → results : `setTimeout(500)`

Ces delais producteurs ne sont pas modifies par cette story — seuls les tests le sont.
**Alternative propre** : attendre le changement d'etat observable (nouvelle question
affichee, nouveau `aria-pressed` groupe, nouvelle route, etc.) via les locators Playwright.

Product Goal lie : le quiz etant le lead magnet central, une suite E2E flaky
bloquerait la velocite de l'equipe sur les prochains sprints (Playwright CI US-15.4
n'a aucun sens si les tests flakient deja en local).

## Fichiers impactes

- `tests/e2e/quiz.spec.ts` — remplacer 2 `waitForTimeout(500)` (lignes 31, 82)
  - Strategie : detecter l'auto-advance via `page.waitForFunction` sur le texte
    de la question, OU attendre la disparition des options actuelles via
    `expect(options.first()).not.toBeVisible({ timeout: 2000 })` suivi d'un
    `expect(options.first()).toBeVisible()` sur les nouvelles options.
  - Alternative : si le composant expose un `data-question-index` sur le container,
    attendre l'increment via `toHaveAttribute`.
- `tests/e2e/accessibility.spec.ts` — remplacer 1 `waitForTimeout(600)` (ligne 68)
  - Strategie : attendre la stabilisation du mobile menu via
    `expect(mobileLinks.last()).toBeVisible()` (deja present ligne 65), puis
    `await page.waitForFunction(() => { const el = document.querySelector('header'); return el && getComputedStyle(el).backgroundColor !== 'rgba(0, 0, 0, 0)'; })`.

## Notes techniques / decisions

### Anti-pattern documente

Playwright propose ces alternatives a `waitForTimeout` :
1. `expect(locator).toBeVisible({ timeout })` — attente jusqu'a visibilite
2. `expect(locator).toHaveText(...)` — attente jusqu'au texte attendu
3. `page.waitForFunction(() => ...)` — attente d'une assertion JS custom
4. `page.waitForURL(...)` — attente d'un changement de route

Le cas d'usage quiz (auto-advance) est **typiquement** resolu par #2 ou #3.

### Performance attendue

Le remplacement devrait **reduire** la duree totale : aujourd'hui le test
attend `500ms` systematiquement meme si l'UI est deja prete en 50ms.
Avec des attentes explicites, Playwright passe des que l'assertion est verte.

### Risque de regression

Faible : le comportement de production (`setTimeout` dans `useQuiz.ts`) reste
inchange. Seule la strategie d'attente cote test est modifiee.

Si un test devient flaky apres le refactoring, la cause sera une mauvaise
selection de locator, pas un probleme de timing absolu.
