## Contexte

### Findings QA 10/04/2026

Le hook `src/hooks/useQuiz.ts` (livre Sprint 7, 150 lignes) est actuellement couvert a **0% — aucun test unitaire**. La seule couverture actuelle est indirecte via les 3 tests E2E Playwright `quiz.spec.ts` qui verifient uniquement le **happy path fonctionnel** en browser.

| Action exportee | Couverte unit ? | Couverte E2E ? |
|---|---|---|
| `setDemographic(field, value)` | ❌ | ✅ (via selectOption) |
| `submitDemographics()` | ❌ | ✅ (via clic "Demarrer") |
| `selectAnswer(id, score)` — auto-advance branch | ❌ | ✅ (3 runs) |
| `selectAnswer(id, score)` — compute results branch | ❌ | ✅ (last question) |
| `goBack()` — questions → questions | ❌ | ❌ |
| `goBack()` — question 0 → demographics | ❌ | ❌ |
| `reset()` | ❌ | ✅ partielle (restart test) |
| `progress` calcule pour 3 phases | ❌ | ❌ |

### Pourquoi c'est critique

1. **Lead magnet central** : ce hook orchestre **tout le funnel quiz** (Product Goal : 50-200 leads/mois). Une regression sur `selectAnswer` casse la completion, donc la capture de lead.
2. **Setters asynchrones (setTimeout 350ms + 500ms)** : les lignes 92 et 112 contiennent des transitions temporisees qui sont une zone de risque classique. Un bug de closure stale (`answers` vs `newAnswers`) a deja ete identifie dans le commit `7d8bc37` ("show results screen after quiz completion instead of blank page") — **preuve historique que le hook a produit une regression silencieuse en prod**.
3. **US-21.7 refactor imminent** : le Sprint 14 prevoit le decoupage de `QuizResultsPreview` en 4 sous-composants. Sans tests unit sur `useQuiz`, un refacto de l'integration peut casser le hook sans signal.
4. **Cost saving vs E2E** : un test unit s'execute en ~5ms. Un test E2E equivalent prend 3-5s. Ajouter 10 tests unit ici fait gagner ~30s a chaque run CI une fois US-15.4 livree.

## Criteres d'acceptation (Gherkin)

### Scenario 1 : Etat initial apres mount

```gherkin
Etant donne ITSM_CONFIG (8 dimensions, 9 questions)
Quand on mount le hook useQuiz(ITSM_CONFIG) via @testing-library/react renderHook
Alors result.current.phase === "demographics"
  ET result.current.currentQuestionIndex === 0
  ET result.current.totalQuestions === 9
  ET result.current.answers egale {}
  ET result.current.demographics egale { companySize:"", industry:"", role:"" }
  ET result.current.results est null
  ET result.current.progress === 0
```

### Scenario 2 : setDemographic met a jour un seul champ

```gherkin
Etant donne le hook mount
Quand on invoque act(() => result.current.setDemographic("companySize", "200_499"))
Alors result.current.demographics.companySize === "200_499"
  ET result.current.demographics.industry === "" (inchange)
  ET result.current.phase reste "demographics"
```

### Scenario 3 : submitDemographics transitionne vers questions

```gherkin
Etant donne le hook mount avec demographics rempli
Quand on invoque act(() => result.current.submitDemographics())
Alors result.current.phase === "questions"
  ET result.current.currentQuestionIndex reste 0
  ET result.current.progress === 10 (debut des questions)
```

### Scenario 4 : selectAnswer auto-advance vers la question suivante (timer 350ms)

```gherkin
Etant donne le hook en phase "questions" avec currentQuestionIndex=0
  ET vi.useFakeTimers() actif
Quand on invoque act(() => result.current.selectAnswer("q1", 3))
  ET on advance les fake timers de 350ms
Alors result.current.answers contient { q1: 3 }
  ET result.current.currentQuestionIndex === 1
  ET result.current.phase reste "questions"
```

### Scenario 5 : selectAnswer sur la derniere question calcule les results

```gherkin
Etant donne le hook avance a currentQuestionIndex === totalQuestions - 1 (= 8)
  ET les 8 questions precedentes ont score 3
  ET vi.useFakeTimers() actif
Quand on invoque act(() => result.current.selectAnswer("q9", 3))
  ET on advance les fake timers de 500ms
Alors result.current.phase === "results"
  ET result.current.results !== null
  ET result.current.results.overallScore est un nombre entre 0 et 100
  ET result.current.results.dimensionScores a 8 cles
  ET result.current.results.weakestDimensions a au maximum 3 items
  ET result.current.results.segment === "itsm"
  ET result.current.progress === 100
```

### Scenario 6 : goBack revient a la question precedente

```gherkin
Etant donne le hook en phase "questions" avec currentQuestionIndex === 2
Quand on invoque act(() => result.current.goBack())
Alors result.current.currentQuestionIndex === 1
  ET result.current.phase reste "questions"
```

### Scenario 7 : goBack depuis question 0 retourne en phase demographics

```gherkin
Etant donne le hook en phase "questions" avec currentQuestionIndex === 0
Quand on invoque act(() => result.current.goBack())
Alors result.current.phase === "demographics"
  ET result.current.currentQuestionIndex reste 0
```

### Scenario 8 : reset remet tous les etats a zero

```gherkin
Etant donne le hook avance a phase "results" avec des answers et results remplis
Quand on invoque act(() => result.current.reset())
Alors result.current.phase === "demographics"
  ET result.current.currentQuestionIndex === 0
  ET result.current.answers egale {}
  ET result.current.demographics egale { companySize:"", industry:"", role:"" }
  ET result.current.results est null
```

### Scenario 9 : progress est coherent sur les 3 phases

```gherkin
Etant donne le hook mount
Quand phase = "demographics"
Alors progress === 0

Et :
Quand phase = "questions" et currentQuestionIndex = 4 et totalQuestions = 9
Alors progress === 10 + round(4/9 * 90) === 50

Et :
Quand phase = "results"
Alors progress === 100
```

### Scenario 10 : selectAnswer ne mute PAS l'etat answers si appele sur meme question multiple fois (idempotence derniere valeur)

```gherkin
Etant donne le hook en phase "questions" avec currentQuestionIndex = 0
Quand on invoque selectAnswer("q1", 2) puis selectAnswer("q1", 4) dans un meme tick
  ET on advance les fake timers de 350ms
Alors result.current.answers.q1 === 4 (derniere valeur gagne)
  ET l'auto-advance se declenche
```

## Notes techniques / decisions

### Pourquoi renderHook et pas mount d'un composant

**Retenu** : `renderHook` de `@testing-library/react`. Avantages :
- Pas de bruit DOM
- Pas de dependance sur une UI qui evoluera (US-21.7 refacto)
- Plus rapide (~5ms vs ~40ms par test)
- Isole le contrat du hook, pas le rendu

**Rejete** : `render(<QuizPage />)` + assertions DOM. Motif : couplage fort avec la UI, tests fragiles au refacto US-21.7.

### Gestion des setTimeout

Le hook utilise `setTimeout(350)` et `setTimeout(500)` pour les animations. **Obligation** d'utiliser `vi.useFakeTimers()` + `vi.advanceTimersByTime(350)` — sinon les tests seraient flaky ou trop lents.

```typescript
it("auto-advances after 350ms", async () => {
  vi.useFakeTimers();
  const { result } = renderHook(() => useQuiz(ITSM_CONFIG));

  act(() => result.current.submitDemographics());
  act(() => result.current.selectAnswer("incident_1", 3));

  expect(result.current.currentQuestionIndex).toBe(0); // pas encore

  act(() => vi.advanceTimersByTime(350));

  expect(result.current.currentQuestionIndex).toBe(1); // advance confirme
});
```

### Piege : closure stale sur answers

Le bug historique `7d8bc37` etait cause par un `setAnswers` qui referencait l'ancien `answers` au lieu du `newAnswers` local. Le test scenario 10 (idempotence) est specifiquement la pour proteger cette zone. Si le hook est refacto pour utiliser `useReducer` un jour, ce scenario reste valide.
