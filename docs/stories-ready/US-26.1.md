```
En tant que developpeur responsable de la securite du site WaS,
je veux upgrader les dependances vulnerables (next 16.2.2 → 16.2.4, next-intl 4.9.0 → 4.9.1, dompurify via npm audit fix),
```

## Contexte

Detection par la tache planifiee `security--accessibility-scan` du 26/04/2026 :
- 🔴 HIGH — `next` 16.2.2 vulnerable a un DoS via Server Components (GHSA-q4gf-8mx6-v5v3)
- 🟡 MOD — `next-intl` 4.9.0 vulnerable a une open redirect (GHSA-8f24-v5vv-gm5j)

- 🟡 MOD — `dompurify` <=3.3.3 — 4 advisories XSS bypass (transitif via `jspdf`, fixable par `npm audit fix`)

Source : `docs/refinement/security-scan-2026-04-26.md` §3.

Scope chirurgical — pas de refactor, pas de nouvelle feature. Upgrade pur + verification regression.

## Notes techniques / decisions

### Strategie upgrade (D38 candidate)

**Pas de `npm audit fix --force`** — risque de breaking change incontrole. Approche progressive :

```bash
# Etape 1 — upgrade explicite des deps directes
npm install next@16.2.4 next-intl@4.9.1

# Etape 2 — audit fix pour les transitifs (sans --force)
npm audit fix

# Etape 3 — verification
npm audit
npm run build
npm run test
npx playwright test
```

### Risque cle : Next.js 16.2.2 → 16.2.4 changelog

**Action obligatoire avant merge** : lire le CHANGELOG Next entre 16.2.2 et 16.2.4 (https://github.com/vercel/next.js/releases). Si un breaking change est detecte → escalade PO immediate + report en S21 avec story de migration dediee.

Cas connu : 16.2.x sont des patch releases de la branche 16.2 — risque tres faible.

### Rollback plan

Si regression detectee post-upgrade :
```bash
git revert <commit-hash>
npm ci  # re-install depuis lockfile prealable
npm run build  # verifier rollback OK
git push
```

Le `npm ci` (et non `npm install`) garantit la restauration exacte du lockfile precedent.

### uuid hors scope

`uuid` <14.0.0 (transitif via `resend` → `svix`) reste vulnerable apres cette story. Il fait l'objet de US-26.3 (veille passive). `npm audit fix --force` proposerait `resend@6.1.3` qui est un downgrade breaking — refuse.

### Pourquoi 2 pts et pas 1

L'upgrade en lui-meme = 30 min. La verification complete (build + 913 tests + 55 E2E + audit + smoke local + smoke preview Vercel + lecture CHANGELOG) + le risque non-nul de regression Next.js justifient 2 pts.

### Pourquoi injecter en S20 (marge) plutot que reporter S21

Vuln HIGH = exposition de 7 jours supplementaires si on attend S21. Manifeste P9 (excellence technique non negociable). La marge S20 (2 pts) etait reservee a une story emergente OU un Refactoring Radar mensuel — celle-ci prime.

**Decision PO requise** : valider l'injection en marge S20 (D39 candidate) avant J1 du sprint.
